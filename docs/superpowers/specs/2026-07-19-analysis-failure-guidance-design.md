# 分析失败诊断与提示词重试设计

## 目标

在分析创建失败时，区分用户可修复的需求/规则问题与服务故障。前者由独立的 LLM 诊断生成可执行建议及改写后的分析需求；后者明确告知可重试，不归因于用户。

## 失败分类

- `ANALYSIS_CONFIG_INVALID`：模型返回不符合报表协议，或规则引用/类型校验失败。保存失败分析，并请求诊断模型。
- `LLM_REQUEST_TIMEOUT`、`LLM_REQUEST_FAILED`：规则生成模型不可用或超时。保存失败分析，不调用诊断模型。
- `D1_WRITE_FAILED`：分析记录、关联资产或回读写入远程 D1 时失败。返回可重试服务错误；不创建第二条失败记录，避免网络结果未知时产生重复记录。

## 诊断模型

诊断调用使用独立、短小的系统提示词，输入仅包含：用户需求、主表名称和字段、失败分类、规则校验问题（以及必要时脱敏的原始模型响应）。它输出严格 JSON：`summary`、`suggestion`、`revisedRequirement`。不得生成报表规则，不得将网络、D1、模型可用性错误归因于用户。

诊断失败不会覆盖原始失败：API 仍返回原始错误及通用重试/修改提示。

## API 与持久化

失败分析记录增加可空字段保存诊断结果和建议后的需求。创建接口在可诊断失败时返回 `guidance`；前端历史记录也能展示该字段。用户点击“应用建议并重新分析”时，将 `revisedRequirement` 填回创建表单，由用户确认后重新提交；不会自动发起第二次分析。

## 可观测性

记录 D1 的主记录写入、资产关联写入、回读三个阶段，以及诊断模型的请求、解析、耗时和失败分类。敏感本地诊断日志沿用现有脱敏开关。

## 测试

- 规则失败返回并保存诊断建议。
- 诊断模型失败不影响原始失败响应。
- D1 写入失败返回 `D1_WRITE_FAILED`，且不落入规则失败/二次失败记录逻辑。
- 前端能展示建议、应用建议到输入框并由用户手动再次提交。
